# Session Journal — 2026-05-02

## Goal

Fix two bugs reported after Sonnet's last session:
1. Booking widget Lambda errors (column subaccount_id does not exist, column color does not exist)
2. Service Calendar bookings not loading

## What we accomplished

### Booking widget + Service Calendar fix
Root cause: Sonnet had built service calendars and 3 booking Lambdas querying `agency_users WHERE subaccount_id=$1` but `agency_users` has NO subaccount_id column. Compounded by ID format mismatch where `services.assigned_staff` and `appointments.assigned_to` held legacy IDs while `subaccount_users` uses UUIDs.

Fixes:
- Migrated subaccount_users schema: added schedule + date_overrides JSONB columns
- One-time migration: 16 appointments + 1 service moved from legacy IDs to UUIDs
- Booking Lambdas patched (booking-widget-data, booking-availability, booking-submit) to query subaccount_users instead of agency_users
- delete-user Lambda built and wired to API Gateway
- update-user Lambda extended to accept color, schedule, date_overrides

### User identity unification (closed Section 3.8 of forward path)
Discovered the larger issue: 5 different sources of user data (subaccount_users table, blob.users, blob.adminProfile, blob._subaccountAdmin, state.user) with different rules for which is canonical. This caused color/schedule edits to land in some places but not others.

Architectural fix:
- subaccount_users is now the SINGLE source of truth
- All UI reads converge on db.users via /api/subaccount/data-load
- All mutations go through /api/subaccount/update-user or /api/subaccount/delete-user
- refreshStateUser() helper added: syncs state.user from db.users at the end of fetchFromServer
- Removed redundant local mutations from saveProfile, saveEditUser, saveUserSched
- One-time blob cleanup: stripped users[], _subaccountAdmin, settings.{adminProfile, supabaseUrl, supabaseKey} from existing blobs
- data-save Lambda hardened to silently strip these fields if ever sent again

### My Profile modal

### Supabase rebrand

Took longest to diagnose. State.user was set at session restore from /api/subaccount/session response which only included id/username/role/name. No color field. Sidebar avatar fell back to default purple. Fixed by adding refreshStateUser() to pull color/schedule from db.users into state.user.

### zsh history expansion
Patrick's shell ate ! characters in regex patterns multiple times. Workaround: write Python scripts to /tmp first, then run them, instead of putting ! in shell command arguments.
