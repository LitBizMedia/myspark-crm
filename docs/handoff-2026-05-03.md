## Open items, ranked by priority

### High priority

**1. CloudWatch alarms (CRITICAL gap for production)**

Currently zero alarms configured. If a Lambda starts erroring, RDS connection pool exhausts, or API Gateway throttles, you find out from a customer complaint, not a notification.

Recommended baseline:
- Lambda error rate > 1% over 5 min, per function
- Lambda duration p99 > 75% of timeout, per function  
- Lambda throttles > 0
- API Gateway 5xx rate > 1%
- RDS CPU > 80%
- RDS connection count > 80% of max
- RDS storage free space < 20%

Effort: 2-3 hours including SNS topic for email alerts.

**2. Bcrypt migration for agency_users**

Agency users still on SHA-256. The `ADMIN_HASH` and `AGENCY_FALLBACK_HASH` constants in `index.html` (lines 1714, 1721) are hardcoded SHA-256 hashes — same hash for both. Anyone reading source has the hash; SHA-256 is fast to brute force.

Path:
1. Add `legacy_password_hash` column to `agency_users` if not present
2. Update agency login Lambda to verify bcrypt first, fall back to legacy SHA-256 on the fly migration (mirror what subaccount login already does)
3. Once all live agency users have logged in once and migrated, drop the legacy column
4. Remove the hardcoded hash constants from `index.html`
5. Replace the client-side fallback path with a server-only break-glass

Effort: 4-6 hours. High security value.

**3. Move AGENCY_FALLBACK_HASH out of plain Lambda env vars**

Currently env vars in plain text on the Lambda. Should be in AWS Secrets Manager with rotation enabled.

Effort: 1 hour.

### Medium priority

**4. Database index audit**

Run this against the DB and verify every multi-tenant table has an index on `subaccount_id`:

```sql
SELECT t.tablename, COUNT(idx.indexname) as idx_count,
       string_agg(idx.indexdef, '; ') as indexes
FROM pg_tables t
LEFT JOIN pg_indexes idx ON idx.tablename = t.tablename
  AND idx.indexdef LIKE '%subaccount_id%'
WHERE t.schemaname = 'public'
GROUP BY t.tablename
ORDER BY t.tablename;
```

Tables that MUST have this index for query performance at scale:
subaccount_data, subaccount_users, subaccount_plans, appointments,
contacts, payments, sessions, audit_log, sms_settings, services,
service_variations, class_sessions, forms, form_submissions,
custom_fields, products, gift_cards, subscriptions, tags.

Effort: 30 min audit + variable execute time depending on what's missing.

### Low priority / cosmetic

**8. Var → let/const refactor**

1664 var declarations in index.html. Modern style is `const` by default, `let` for reassignment. Mechanical refactor with low risk if done with linter help, but ~2 hours and no functional benefit.

**9. Inline event handlers**

725 `onclick=`/`onchange=` etc. inline in HTML. Modern style separates JS from HTML via addEventListener. Refactor would be substantial (~8 hours) and risk-prone. Not urgent.

**10. Inline styles**

1554 inline `style=` attributes. Some are dynamic (e.g., user color), some are static and could move to CSS classes. Refactor benefit: maintainability. Not urgent.

**11. document.write in gift card popup**

Single use (line ~11740) for printing gift cards. The pattern is `window.open()` + `w.document.write(html)` for a NEW popup window — not the dangerous "write to current page" anti-pattern. Could modernize to blob URL but real risk is near zero.

## Architecture state for future sessions

When you start a new session, the next Thalos should know:

- Single source of truth for users: `subaccount_users` table (UUID-keyed)
- For agency users: `agency_users` table (separate, also UUID-keyed)
- Subaccount data blob (`subaccount_data.data` JSONB) holds non-user data only:
  contacts, appointments, tasks, payments, settings (without user fields), etc.
- `data-save` Lambda strips `users[]`, `_subaccountAdmin`,
  `settings.adminProfile`, `settings.supabaseUrl`, `settings.supabaseKey`
  defensively on every save
- All user mutations route through `update-user`, `delete-user` (subaccount-side)
  or `subaccount-admin-update`, `subaccount-admin-get` (agency-side)
- Frontend `refreshStateUser()` syncs `state.user` from `db.users` (canonical)
  at end of `fetchFromServer()`
- Login flow honors `must_change_password` and forces password rotation before
  the user lands in the workspace
- Legacy synthetic IDs (`admin-patrick`, `admin-<slug>`) still recognized by
  `isPrimaryAdmin` for backwards compat with the agency login-as flow's
  fallback path. Can be removed once verified that no localStorage caches
  reference them.
